# ユニット③ auth-user-audit — Functional Design プラン

**作成日**: 2026-07-19
**ユニット**: unit-03-auth-user-audit(認証・ユーザ管理・監査基盤)— Phase 1
**担当ストーリー**: US-001〜009(登録・認証)、US-042(認証イベント記録)、US-045(セキュリティアラートメール)、US-047/048(言語・テーマ切替の本実装)— 13 件

## 1. コンテキスト(確定済み事項)

- パッケージ: `auth` / `user` / `audit` + `common`(メール・i18n・セキュリティ設定・Problem Details・DbDialect)— Application Design 確定
- トークン: アクセス 10 分 / リフレッシュ 24 時間(設定可能)、sessionStorage 保持、ローテーション + 再利用検知(ファミリ一括失効)、リフレッシュトークンはハッシュ化保存
- ロックアウト: D-11(閾値・時間は設定可能、管理者手動解除あり)
- 監査: D-20(同期 Spring イベントリスナー + REQUIRES_NEW、例外パスでも発行)、H-07(イベント種別コード + パラメータで記録)
- メール: MailService(テンプレート + 受信者言語 — D-09)、US-001 は申請時 UI 言語、既存ユーザは設定言語。開発は MailPit
- 初期管理者: 環境変数からブートストラップ(US-004、重複作成なし)
- 列挙攻撃対策: US-001 のレスポンスは登録済み/未登録で区別不能
- CSP 本則・セキュリティヘッダーの導入は③スコープ(②からの申し送り: `script-src 'self'` で成立)

## 3. 確認事項(回答をお願いします)

**Q1. 内部 DB の主キー戦略**
ユーザ・トークン・監査ログ等の内部テーブルの ID 形式は?

- A) BIGINT 自動採番(IDENTITY)— シンプル・インデックス効率良、内部 DB 完結なので衝突リスクなし(推奨)
- B) UUID(v7)— 分散・露出耐性はあるがサイズ増。API 露出 ID は別途対策可能
- C) その他

[Answer]: A(BIGINT 自動採番)

**Q2. セキュリティアラート(US-045)の閾値評価方式**

- A) 固定時間窓 — 「直近 N 分間に失敗 M 件以上」で通知、同一窓内は再通知しない(クールダウン)。実装が単純で誤報も抑えられる(推奨)
- B) 連続失敗回数の累積のみ(時間窓なし)
- C) その他

[Answer]: A(固定時間窓 + クールダウン)

**Q3. 言語・テーマ設定のサーバ保存範囲(US-047/048 の本実装)**
メール言語(受信者の言語設定)のため、少なくとも言語はユーザ属性としてサーバ保存が必要です。

- A) 言語・テーマとも user のユーザ設定としてサーバ保存(ログイン後はサーバ値を正、localStorage は未ログイン時・初期表示用)— 挙動が一貫(推奨)
- B) 言語のみサーバ保存、テーマは localStorage のまま(テーマは端末ごとで良いという考え方)
- C) その他

[Answer]: A(言語・テーマともサーバ保存。ログイン後はサーバ値が正)

**Q4. メール送信の実行方式**

- A) 同期送信(リクエスト処理内)。ただし US-001 系は送信失敗でもレスポンスを変えず(列挙対策維持)、失敗は監査ログに記録 — MVP としてシンプル(推奨)
- B) 非同期送信(@Async + 再試行)— レスポンス速度は安定するが失敗時の追跡・再試行設計が増える
- C) その他

[Answer]: A(同期送信。列挙対策維持・失敗は監査記録)
