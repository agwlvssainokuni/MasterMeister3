# ユニット④ connection-permission — Functional Design プラン

**作成日**: 2026-07-19
**ユニット**: unit-04-connection-permission(接続・スキーマ取込・権限)— Phase 2
**担当ストーリー**: US-010/011(接続管理)、US-012(スキーマ取込)、US-013〜015(権限・実効権限解決)、US-016/017(YAML)、US-018/019(グループ)、US-043(管理操作の監査記録)— 11 件

## 1. コンテキスト(確定済み事項)

- パッケージ: `connection` / `metadata` / `permission`(Application Design 確定)。主要クラス: ConnectionService / CredentialEncryptor / TargetDataSourceRegistry、SchemaImportService / MetadataQueryService、PermissionService / EffectivePermissionResolver / GroupService / PermissionYamlService
- 接続パスワードは可逆暗号化で内部 DB 保存、鍵は環境変数供給(H-03)。鍵ローテーション時の再暗号化手順を設計で定義する
- 対象 RDBMS は MySQL / MariaDB / PostgreSQL / H2 の 4 種。コネクションプール必須(DriverManager 直接使用は不可)
- 権限モデル: 主権限 NONE / READ / UPDATE(スキーマ/テーブル/カラム 3 階層)+ 補助権限 CREATE / DELETE(スキーマ/テーブル単位の真偽値)。「未設定」と「明示的 NONE」は別状態(D-18)
- 実効権限解決は D-21 アルゴリズム(個別チェーン優先 → グループ合成(許可的優先・OR)→ デフォルト拒否)。確定例 2 件(US-015)あり。Caffeine キャッシュ + 粗粒度無効化(権限変更・グループ変更・スキーマ再取込)
- YAML インポートは全置換方式・重複エントリは全体拒否・単一トランザクション(US-017)
- グループ削除は所属関係・権限設定のカスケード削除(US-018)
- 管理操作(接続変更・取込・権限変更・グループ変更・YAML)はそれぞれ独立イベント種別で監査記録(US-043、H-07 コード記録)— ③の AuditEventPublisher + 種別拡張規約(名詞_過去分詞)を使用
- PBT ブロッキング対象: EffectivePermissionResolver の不変条件(PBT-03)、PermissionYamlService のラウンドトリップ(PBT-02)
- API・認可は③の基盤に従う: /api/admin/** = ADMIN、Problem Details、AppProperties(mm.app.*)拡張パターン

**Q1. 接続パスワード暗号化(H-03)の方式と鍵ローテーション対応範囲**

A) AES-256-GCM、鍵は環境変数(Base64)で供給。保存形式に鍵 ID を含め、設定では現行鍵 + 旧鍵の複数を受け付ける。旧鍵で復号できた資格情報は次回保存時に現行鍵で再暗号化(段階的ローテーション)。一括再暗号化の管理操作は将来スコープとし手順のみ設計書に定義(推奨)

B) AES-256-GCM、単一鍵のみ。鍵ローテーションは「新しい鍵に切替後、全接続のパスワードを再入力」という運用手順の文書化で対応(実装は最小)

**Q2. スキーマ再取込で存在しなくなった対象を参照する権限エントリの扱い**

権限エントリは物理名(スキーマ/テーブル/カラムの文字列)で保持します。再取込後にメタデータ側に存在しなくなった対象を指すエントリ(孤児エントリ)の扱いは?

A) エントリは保持する。実効権限解決は取込済みメタデータに存在するカラムに対してのみ行うため孤児エントリは無害。権限画面で「対象なし」と注記表示し、削除は管理者の明示操作(推奨 — テーブルの一時退避・再作成で権限設定が消えない)

B) 再取込時に孤児エントリを自動削除する(メタデータと権限が常に整合。ただし再取込のたびに権限が消えるリスク)

**Q3. YAML のプリンシパル表現と未知プリンシパルの扱い(US-016/017)**

A) ユーザは email、グループはグループ名で表現(人が読め、環境間移行しやすい)。インポート時に存在しない email / グループ名が 1 件でもあれば全体拒否(重複拒否と同じ扱い — US-017 の「部分適用なし」と一貫)(推奨)

B) ユーザは email、グループはグループ名で表現。未知プリンシパルのエントリはスキップして残りを適用し、スキップ内容を結果レポートで返す

C) ユーザ・グループとも内部 ID で表現(同一環境のバックアップ/リストア専用と割り切る)

**Q4. 接続プールのライフサイクルと疎通確認(US-010/011)**

A) プールは遅延生成 + キャッシュ(初回利用時に生成、接続設定の変更・削除で破棄)。接続の登録・編集画面に「接続テスト」操作を設ける(保存とは独立に疎通確認、結果は監査記録)(推奨)

B) プールは遅延生成 + キャッシュのみで、接続テスト機能は設けない(スキーマ取込の成否が実質の疎通確認)

**Q5. 権限設定画面の編集方式(US-013/014)**

A) プリンシパル軸: 接続 → ユーザまたはグループを選択 → スキーマ/テーブル/カラムのツリーを表示し、階層ごとに主権限(未設定 / NONE / READ / UPDATE)と補助権限(CREATE / DELETE)を設定(推奨 — D-21 の「プリンシパルごとのチェーン」と一致し実効権限を予測しやすい)

C) プリンシパル軸で編集し、リソース軸は参照専用ビュー(どのプリンシパルに設定があるかの確認用)を追加
