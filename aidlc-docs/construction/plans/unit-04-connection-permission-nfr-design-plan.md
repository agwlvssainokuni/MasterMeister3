## 確認事項(回答をお願いします)

**Q1. 対象 RDBMS への接続・操作が失敗したときのリトライ方針**

A) リトライしない(fail-fast)— 失敗を即時に理由コード付きで返す。④の操作(接続テスト・取込・管理操作)はすべて管理者の対話操作であり、明示的な再実行が自然。Resiliency Baseline 無効の方針とも整合(推奨)

B) 1 回だけ自動リトライ(接続確立失敗時のみ)— 一過性のネットワーク断には強いが、失敗応答が遅くなり原因も見えにくい

C) その他(下の [Answer]: の後に記述してください)

[Answer]: A(リトライせず即エラー — fail-fast)

**Q2. プール枯渇(pool_timeout_ms 内に接続を取得できない)時の API 応答**

A) 502 + 理由コード `TIMEOUT` — スキーマ取込失敗(接続不能)と同じ「上流 DB 起因の失敗」として統一的に扱う(推奨)

B) 503(Service Unavailable)+ Retry-After — 「一時的な混雑」として区別する。ただし④の管理操作では混雑が起きにくく、区別の価値は⑤⑥(データアクセス)で再評価でも良い

C) その他(下の [Answer]: の後に記述してください)

[Answer]: B(503 + Retry-After — 一時的混雑を上流障害 502 と区別。⑤⑥のデータアクセスにも同じ規約を適用)

**Q3. 登録可能な接続数の上限**

A) 上限を設けない — 接続数は管理者の運用判断(対象 RDBMS 側リソースへの影響はプール設定で個別制御)。実装も画面もシンプル(推奨)

B) 設定可能な上限(`mm.app.connection.max-count`、既定例 20)— プール総数の暴走を構成レベルで防ぐ

C) その他(下の [Answer]: の後に記述してください)

[Answer]: A(上限なし — 接続数は管理者の運用判断)

**Q4. スキーマ取込のメタデータ保存方式**

A) JPA エンティティ + Hibernate バッチ INSERT(batch_size 100、order_inserts)— エンティティ定義を参照系と共用でき、コードが一貫。2,000 テーブル規模なら十分な速度(推奨)

B) JdbcTemplate のバッチ INSERT — 最速だが INSERT 文の手書きが増え、エンティティとの二重管理になる

C) その他(下の [Answer]: の後に記述してください)

[Answer]: C(シンプルに saveAll — バッチ最適化なし。PK が IDENTITY のため Hibernate バッチは効かず、内部 H2 は同一プロセスで往復コストがなく、取込は低頻度の管理者操作のため。実測で問題が出たら SEQUENCE 化 + バッチを検討 — ユーザ提案 2026-07-19)

**Q5. Testcontainers 結合テストの範囲(3 エンジン共通)**

A) 最小 2 系統 — (a) 接続テスト成功、(b) サンプルスキーマの取込結果検証。方言依存はこの 2 つに集中しており、権限解決・YAML は方言非依存のため H2 で網羅(実行時間を抑制)(推奨)

B) 拡大 — 上記に加え、実エンジン上で取込 → 権限設定 → 実効権限解決 → YAML 往復の一連も 3 エンジンで実行(網羅性は上がるが CI 時間が増え、方言非依存部の重複実行になる)

C) その他(下の [Answer]: の後に記述してください)

[Answer]: A(最小 2 系統 × 3 エンジン = 6 テスト。コンテナはエンジンごと 1 個・計 3 個 — ユーザ確認済み)

## B. 質問にしない実装確定事項(Security — 根拠付き)

- YAML ボディ上限 1 MB は**アプリケーション内検証**で実施(Tomcat の maxPostSize はフォーム POST 限定で JSON/YAML ボディに効かないため。リバースプロキシ前提にはしない — 単体 WAR で成立させる)
