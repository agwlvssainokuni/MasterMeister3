# ユニット定義(Unit of Work)— MasterMeister3

**作成日**: 2026-07-18
**前提**: デプロイ形態は単一実行可能 WAR のモノリス(D-14)。各ユニットは独立デプロイのサービスではなく、**直列に開発する作業単位(モジュール)**である(ユニットプラン Q1=A, Q2=A)。

## 1. ユニット一覧

| # | ユニット名 | 対応フェーズ | 概要 |
|---|---|---|---|
| ① | foundation(骨格・開発環境) | 準備 | ビルドパイプライン貫通と開発環境整備 |
| ② | design-system(デザインシステム+モック) | 準備 | フルスクラッチのデザインシステムと画面モック(承認ゲート — D-08) |
| ③ | auth-user-audit(認証・ユーザ管理・監査基盤) | Phase 1 | ユーザ登録・承認、JWT 認証、監査記録基盤、UI 基盤統合 |
| ④ | connection-permission(接続・スキーマ取込・権限) | Phase 2 | 対象 RDBMS 接続、スキーマ取込、権限・グループ・YAML |
| ⑤ | record-maintenance(マスタメンテナンス) | Phase 3 | テーブル/レコードの閲覧・編集・一括反映 |
| ⑥ | query(クエリ系+監査閲覧) | Phase 4 | クエリビルダー・保存・実行・履歴、監査ログ閲覧 |

## 2. 各ユニットの責務と完了条件

### ① foundation(骨格・開発環境)
- **責務**: ルート Gradle マルチプロジェクト、Spring Boot 空アプリ(backend サブプロジェクト)、React+TypeScript+Vite 雛形、内部 DB(H2)+ Flyway 骨格、docker compose による対象 RDBMS 開発環境(MySQL/MariaDB/PostgreSQL。H2 は内蔵)
- **完了条件(DoD)**: `./gradlew build` 一発で npm/Vite ビルドが実行され、**空の React 画面を配信する実行可能 WAR** が得られる(ユニットプラン Q4=A 補足確定)。docker compose で対象 RDBMS 3 種が起動する
- **対応ストーリー**: なし(基盤整備。全ストーリーは②〜⑥に割当)

### ② design-system(デザインシステム+モック)
- **責務**: デザイントークン(CSS カスタムプロパティ、ライト/ダーク — D-10)、共有 UI コンポーネント、主要画面のモックカタログ(Vite ページ — D-07/D-08)。言語切替・テーマ切替の UI 実装を先行(US-047/048 の 2 段構えの前段)
- **完了条件(DoD)**: モックページ群でトークン・コンポーネント・主要画面イメージ(テーマ/言語切替含む)を確認し、**ユーザのモック承認ゲート(D-08)を通過**する。モックはそのまま実装基盤となる(使い捨てにしない)
- **対応ストーリー**: US-047/048 の UI 部分を先行実装(主割当は③)

### ③ auth-user-audit(認証・ユーザ管理・監査基盤)— Phase 1
- **責務**: バックエンド `auth` / `user` / `audit` パッケージ + `common`(メール・i18n・セキュリティ設定・Problem Details)、フロントエンド認証画面・管理画面骨格。監査記録基盤(D-20: 別トランザクション)は**以後の全ユニットが呼ぶためここで構築**。UI 基盤(言語・テーマ)の本アプリ統合
- **完了条件(DoD)**: 登録申請→承認→ログイン→リフレッシュ→ログアウトの一連が動作し、認証・セキュリティイベントが監査記録され、閾値超過でアラートメールが送信される(D-13)。UI 言語・テーマ切替がアプリ上で動作する
- **対応ストーリー**: US-001〜009、US-042、US-045、US-047、US-048(13 件)

### ④ connection-permission(接続・スキーマ取込・権限)— Phase 2
- **責務**: `connection` / `metadata` / `permission` パッケージ。接続管理(暗号化 — H-03)、スキーマ取込、権限エントリ(D-18)・実効権限解決(D-21 + キャッシュ)・グループ・YAML(US-016/017)。管理操作の監査記録
- **完了条件(DoD)**: 接続登録→スキーマ取込→権限設定(個別/グループ/YAML)の一連が動作し、実効権限解決が確定例(US-015 の 2 例)を含めて正しい。PBT: `EffectivePermissionResolver`(PBT-03)、`PermissionYamlService`(PBT-02)が成立
- **対応ストーリー**: US-010〜019、US-043(11 件)

### ⑤ record-maintenance(マスタメンテナンス)— Phase 3
- **責務**: `record` パッケージ + レコード一覧・編集 UI(編集バッファ = useReducer)。実効権限によるカラムフィルタ(NONE 完全非表示)、カラムメタデータ同梱応答、一括反映(オールオアナッシング — US-027)、データアクセス監査
- **完了条件(DoD)**: READ/UPDATE 権限に応じた閲覧・編集・作成・削除・一括反映が対象 RDBMS 4 方言で動作する。主キーなしテーブルの制限(D-15)が機能する
- **対応ストーリー**: US-020〜027、US-044(9 件。US-044 の主割当は⑤、クエリ実行分の記録は⑥で拡張)

### ⑥ query(クエリ系+監査閲覧)— Phase 4
- **責務**: `query` パッケージ + クエリビルダー/実行/履歴 UI、監査ログ閲覧画面(US-046 — ユニットプラン Q3=A)。読み取り専用強制(H-01)、スキーマレベル検証(D-16)、方言別スキーマ切替
- **完了条件(DoD)**: ビルダー⇄SQL の相互変換、保存クエリ、パラメータ付き実行、履歴、監査閲覧が動作する。PBT: `QueryBuilderService.buildSql/parseSql`(PBT-02)が成立
- **対応ストーリー**: US-028〜041、US-046(15 件)

## 4. コード編成方針(グリーンフィールド)

要件 §6 で確定したルート Gradle マルチプロジェクト構成に、全ユニットが積み上げる(ユニットごとに新リポジトリ・新プロジェクトは作らない):

```
<WORKSPACE-ROOT>/
├── settings.gradle.kts / gradlew     # ユニット①で作成
├── backend/                          # Spring Boot(機能パッケージ x8 + common)
│   └── src/main/java/...            # ユニット③〜⑥で機能パッケージを追加
├── frontend/                         # React + TypeScript + Vite
│   └── src/
│       ├── design-system/           # ユニット②で作成(モック含む)
│       └── features/                # ユニット③〜⑥で画面を追加
├── devenv/                           # docker compose(対象 RDBMS)— ユニット①
└── aidlc-docs/                       # ドキュメントのみ(コード禁止)
```
