## 2. プロジェクト概要

- RDBMS に格納されたマスタデータをメンテナンスする Web アプリケーション
- **開発体制**: 単独開発者による初期開発
- **開発方針**: MVP ファーストで段階的に機能を追加
- **実装優先順位**: ユーザ管理 → 対象 RDBMS セットアップ → アクセス制御 → データ表示(ワークフロー上流から)

| D-21 | 実効権限はカラムごとに次の順で解決する: (1) ユーザ個別のチェーン(カラム→テーブル→スキーマ)に明示エントリがあればその値(グループは参照しない)、(2) なければ各グループのチェーンを解決し許可的な方を合成、(3) どこにもなければアクセス権なし。具体例 2 件をユーザと確認済み(stories.md US-015 参照) | ストーリーレビュー時のユーザ確認(2026-07-18、解釈(a)) |

## 6. プロジェクト構成(ワークスペースレイアウト)

ルート Gradle マルチプロジェクト統合方式(Q14=B、D-14)。project-overview.md §4 の 3 ディレクトリ構成を維持しつつ、ビルドを統合する。

```
MasterMeister3/
+-- settings.gradle.kts     # ルート: backend をサブプロジェクトとして include
+-- gradlew / gradle/       # Gradle wrapper(ルートに配置)
+-- backend/                # Spring Boot アプリケーション(Gradle サブプロジェクト)
+-- frontend/               # React アプリケーション(Vite。npm 管理は独立)
+-- devenv/                 # 開発環境 (Docker Compose)
+-- aidlc-docs/             # AI-DLC ドキュメント
```

- `./gradlew build` の単一コマンドで npm install → Vite ビルド → dist/ の WAR 静的リソース組み込み → 実行可能 WAR 生成まで完結する
- 開発時は Vite dev サーバ(バックエンドへのプロキシ)と bootRun を並行起動する

## 9. 設計ステージへの申し送り事項

要件レベルでは確定不要だが、設計時に必ず扱うべき注意点。

| ID | 事項 | 内容 |
|---|---|---|
| H-01 | 読み取り専用 SQL の強制方式 | キーワードフィルタでは不十分(CTE、DB 方言、関数の副作用)。読み取り専用トランザクション/接続(`Connection.setReadOnly`、`SET TRANSACTION READ ONLY` 等の方言別実装)を併用する |
| H-02 | 内部 DB のスキーマ移行 | 内部 DB(H2)のスキーマ管理にマイグレーションツール(Flyway 等)を導入し、バージョンアップ時のデータ引き継ぎを担保する |
| H-03 | 接続パスワード暗号鍵の管理 | 可逆暗号化の鍵は環境変数で供給(12-Factor / SECURITY-12)。鍵ローテーション時の再暗号化手順を設計で定義する |
| H-04 | 非対応カラム型のフォールバック | 4 RDBMS の特殊型(BLOB、JSON、配列、空間型等)は MVP では編集対象外とし、読み取り専用表示等のフォールバックを定義する |
| H-05 | 同時編集の競合制御 | レコード編集の楽観ロック方式(取得時値との比較等)を設計で定義する。オールオアナッシング反映(§5.4)との整合を取る |
| H-06 | 開発時 CSP の緩和 | Vite dev サーバ(HMR)はインラインスクリプトを要するため、CSP は本番プロファイルで厳格化し、開発プロファイルのみ緩和する(SECURITY-04 の正当化文書に記載) |
| H-07 | 監査ログのロケール非依存 | 監査ログにはメッセージ文字列ではなくイベント種別コード・パラメータを記録し、表示時に UI 言語で解決する(i18n D-09 との整合) |
