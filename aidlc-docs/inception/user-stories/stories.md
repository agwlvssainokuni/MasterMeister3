## Epic 1: ユーザ登録(FR-01)— Phase 1

### US-001: メールアドレスで登録を申請する
**ユウキ(登録前)** として、メールアドレスを送信して登録を開始したい。パスワードや個人情報を最初から入力せずに始められるからだ。

- Given 未登録のメールアドレス / When 登録申請を送信する / Then 有効期限付きリンクを含む登録確認メールが送信される
- Given 登録済みのメールアドレス / When 登録申請を送信する / Then メールは送信されないが、API レスポンスは未登録の場合と区別できない(列挙攻撃対策)
- Given 登録申請 / When トークンが発行される / Then 有効期限は設定値(`mm.app.user-registration.token-expiry`、デフォルト 3 時間)に従う

### US-002: パスワードを設定して登録を完了する
**ユウキ(登録前)** として、メール内のリンクからパスワードを設定して登録を完了したい。

- Given 有効な登録トークンを含むリンク / When パスワードを設定する / Then 登録が完了し、管理者ダッシュボードの一覧に表示される
- Given 有効期限切れのトークン / When リンクを開く / Then 登録は完了せず、期限切れである旨が表示される
- Given 使用済みのトークン / When 再度リンクを開く / Then 登録処理は実行されない
- パスワードは 8 文字以上を必須とし、アダプティブハッシュで保存される(SECURITY-12)

### US-003: 登録ユーザを承認・却下する
**アキラ** として、登録完了ユーザを承認または却下したい。利用者を統制下に置くためだ。

- Given 登録完了ユーザの一覧 / When 承認する / Then ユーザはログイン可能になり、承認メールが送信される
- Given 登録完了ユーザ / When 却下する / Then ユーザはログイン不可のままで、却下メールが送信される
- Given 未承認ユーザ / When ログインを試みる / Then 認証は拒否される

### US-004: 初期管理者アカウントを自動作成する
**アキラ** として、アプリ初回起動時に環境変数から初期管理者アカウントが自動作成されてほしい。登録フローなしで運用を開始するためだ。

- Given 環境変数 `mm.app.admin.bootstrap.email` / `mm.app.admin.bootstrap.password` が設定された初回起動 / When アプリが起動する / Then 管理者アカウントが作成され、通常の登録フローを経ずにログイン可能
- Given 同一メールアドレスの管理者が既に存在する / When アプリが起動する / Then 重複作成しない

## Epic 2: ユーザ認証(FR-07)— Phase 1

### US-005: ログインする
**ユウキ** として、メールアドレスとパスワードでログインしたい。

- Given 承認済みユーザの正しい資格情報 / When ログインする / Then アクセストークン(デフォルト 10 分)とリフレッシュトークン(デフォルト 24 時間)が発行される
- Given 誤った資格情報 / When ログインする / Then 認証は拒否され、失敗が監査ログに記録される
- 有効期限は `mm.app.jwt.access-token-expiry` / `mm.app.jwt.refresh-token-expiry` で設定可能

### US-006: アカウントロックアウトで保護される(D-11)
**ユウキ** として、ログイン試行が繰り返し失敗した場合はアカウントがロックされてほしい。第三者の総当たり攻撃から守られるためだ。

- Given 連続ログイン失敗が閾値(環境変数で設定可能)に達した / When さらにログインを試みる / Then 正しいパスワードでも一定時間(設定可能)ログインできない
- Given ロック中のアカウント / When ロック時間が経過する / Then 再度ログイン可能になる
- Given ロック中のアカウント / When 管理者が手動解除する / Then 即時にログイン可能になる

### US-007: トークンをリフレッシュする
**ユウキ** として、作業中はログインし直さずにセッションを継続したい。

- Given 有効なリフレッシュトークン / When `/api/auth/refresh` を呼ぶ / Then 新しいアクセストークンと新しいリフレッシュトークンが発行され、旧リフレッシュトークンは無効化される(ローテーション、1 回限り)
- リフレッシュトークンは内部 DB にハッシュ化して保存され、平文は保持されない

### US-008: トークン再利用を検知して防御する
**アキラ** として、無効化済みリフレッシュトークンの再提示があった場合、同系列の全トークンを失効させてほしい。トークン窃取の被害を最小化するためだ。

- Given 使用済み(無効化済み)のリフレッシュトークン / When それが再度提示される / Then 同じトークンファミリに属する全トークンが一括失効し、イベントが監査ログに記録される

### US-009: ログアウトする
**ユウキ** として、ログアウトしてセッションを終了したい。

- Given ログイン中 / When `/api/auth/logout` を呼ぶ / Then リフレッシュトークンが無効化され、以後のリフレッシュは拒否される

## Epic 3: 対象 RDBMS 接続管理(FR-02)— Phase 2

### US-010: 対象 RDBMS 接続を登録する
**アキラ** として、対象 RDBMS の接続情報を登録したい。メンテナンス対象のデータベースをアプリに接続するためだ。

- Given 管理者ダッシュボード / When 接続情報(ホスト、ポート、DB 種別、認証情報、プール設定)を登録する / Then 接続は独立した ID を持ち一覧に表示される
- Given 接続パスワード / When 保存される / Then 内部 DB に可逆暗号化(鍵は環境変数供給)で保存され、平文では保存されない
- MySQL / MariaDB / PostgreSQL / H2 の 4 種をサポートする
- 接続には DB コネクションプールを使用する(DriverManager 直接使用は不可)

### US-011: 接続を複数管理する
**アキラ** として、複数の対象 RDBMS 接続を登録・編集・削除したい。

- Given 複数の接続が登録済み / When 一覧を表示する / Then 各接続が独立して表示され、取込スキーマ・権限設定・監査ログは接続 ID ごとに分離されている
- Given 接続設定の変更 / When 保存する / Then 変更が監査ログに記録される

## Epic 4: スキーマ取込(FR-03)— Phase 2

### US-012: スキーマを取り込む
**アキラ** として、対象 RDBMS からテーブル/ビュー構造を取り込みたい。権限設定と画面表示の基礎データにするためだ。

- Given 登録済みの接続 / When スキーマ更新操作を実行する / Then テーブル/ビューの物理名・コメント・型・制約が内部 DB に接続 ID 単位で取り込まれる
- Given 取込の実行 / When 完了する / Then 操作と結果(成功/失敗)が監査ログに記録される
- Given 再取込 / When 完了する / Then 実効権限キャッシュが無効化される

## Epic 5: アクセス制御(FR-04)— Phase 2

### US-013: 主権限を設定する
**アキラ** として、スキーマ/テーブル/カラムの各階層に主権限(NONE / READ / UPDATE)を設定したい。

- Given 取込済みスキーマ / When ユーザまたはグループに主権限を設定する / Then スキーマ/テーブル/カラムの任意の階層に 3 値のいずれかを設定できる
- Given 複数階層に設定がある / When 実効権限を解決する / Then より詳細な階層(カラム > テーブル > スキーマ)の設定が優先する
- Given 設定済みの権限エントリ / When 権限設定を削除する / Then エントリのない状態(未設定)に戻り、当該階層は上位階層の設定に従う — **明示的な NONE 設定とは別の操作**(D-18)
- Given 明示的 NONE が詳細階層に設定されている / When 実効権限を解決する / Then 上位階層がより許可的でも NONE が優先する(詳細階層優先の原則どおり)
- Given 権限設定の変更(設定・変更・削除いずれも)/ When 保存する / Then 監査ログに記録され、実効権限キャッシュが無効化される

### US-014: 補助権限を設定する
**アキラ** として、スキーマ/テーブル単位に CREATE / DELETE の補助権限を設定したい。

- Given 取込済みスキーマ / When 補助権限を設定する / Then スキーマ/テーブル単位の独立した真偽値として保存される
- Given `CREATE` 付与 + 主キーなしテーブル、または全主キー列が UPDATE 以上 / When 作成可否を判定する / Then レコード作成可能
- Given `DELETE` 付与 + 主キーあり + 全主キー列が READ 以上 / When 削除可否を判定する / Then レコード削除可能
- Given 主キーを持たないテーブル / When 削除可否を判定する / Then `DELETE` 付与の有無によらず常に削除不可(D-15: UPDATE も不可)

### US-015: グループ権限と個別権限を合成する
**ユウキ** として、所属グループと自分個別の権限設定から一貫した実効権限が解決されてほしい。どの画面でも同じ権限で扱われるためだ。

**解決アルゴリズム(D-21、カラムごとに評価)**:
1. ユーザ個別のチェーン(カラム → テーブル → スキーマ)を探索し、明示エントリが見つかればその値が実効権限(このカラムについてグループ設定は参照しない)
2. 個別チェーンに明示エントリがなければ、各グループのチェーンを同様に解決し、より許可的な値(主権限は上位、補助権限は OR)を合成して採用
3. どこにも明示エントリがなければアクセス権なし(デフォルト拒否。明示的 NONE とは別状態 — D-18)

- Given ユーザ個別 = スキーマに READ、グループ = あるカラムに UPDATE / When 解決する / Then 当該ユーザはスキーマ内の**全カラムが READ**(グループのカラム UPDATE は適用されない)。個別設定を持たない他のグループメンバーは当該カラムのみ UPDATE、それ以外(他テーブルのカラム含む)はアクセス権なし(ユーザ確認済みの具体例 1)
- Given ユーザ個別 = あるカラムに READ、グループ = スキーマに UPDATE / When 解決する / Then 当該ユーザは**そのカラムのみ READ**、スキーマ内の他の全カラム(他テーブル含む)は UPDATE。個別設定を持たない他のグループメンバーは全カラム UPDATE(ユーザ確認済みの具体例 2)
- Given 解決済みの実効権限 / When 同じ解決を再要求する / Then キャッシュ(Caffeine)から返される。権限変更・グループ構成変更・スキーマ再取込のいずれかで無効化される
- **PBT 対象(PBT-03)**: 本アルゴリズムの不変条件(個別エントリ存在時のグループ非参照、グループ合成の単調性 = グループ追加で権限が狭まらない、キャッシュ有無での結果同一性)を主要対象とする

## Epic 6: 権限の YAML エクスポート/インポート(FR-05)— Phase 2

### US-016: 権限設定を YAML でエクスポートする
**アキラ** として、接続単位の権限設定を YAML でエクスポートしたい。バックアップや環境間移行のためだ。

- Given 権限設定済みの接続 / When エクスポートする / Then 権限設定一式が YAML 形式でダウンロードでき、操作が監査ログに記録される(独立したイベント種別)

### US-017: 権限設定を YAML でインポートする
**アキラ** として、YAML から権限設定を復元したい。

- Given 有効な YAML / When インポートする / Then 対象接続の既存権限をすべて削除したうえで YAML の内容から再構築される(全置換方式)
- Given 重複エントリ(同一プリンシパル×スキーマ×テーブル×カラム、または同一プリンシパル×スキーマ×テーブル×補助権限種別)を含む YAML / When インポートする / Then インポート全体が拒否され、部分適用は発生しない(単一トランザクション)
- Given インポートの実行 / When 完了する / Then 操作が監査ログに記録される(独立したイベント種別)

## Epic 7: グループ管理(FR-06)— Phase 2

### US-018: グループを作成・改名・削除する
**アキラ** として、ユーザグループを管理したい。部署単位などで権限をまとめて設定するためだ。

- Given 管理者ダッシュボード / When グループを作成・改名する / Then 変更が反映され監査ログに記録される
- Given グループの削除 / When 実行する / Then 所属関係と当該グループの権限設定もあわせて削除される(カスケード)し、実効権限キャッシュが無効化される

### US-019: グループのメンバーを管理する
**アキラ** として、グループへのユーザ追加・削除を行いたい。

- Given グループとユーザ / When 追加または削除する / Then 所属が変更され、対象ユーザの実効権限キャッシュが無効化され、変更が監査ログに記録される

## Epic 8: マスタメンテナンス(FR-08)— Phase 3

### US-020: アクセス可能なテーブル/ビュー一覧を見る
**ユウキ** として、トップ画面で自分がアクセスできるテーブル/ビューの一覧を見たい。

- Given ログイン済み + 接続とスキーマを選択 / When トップ画面を表示する / Then 実効権限が READ 以上のテーブル/ビューのみが表示される
- Given アクセス可能なテーブルがない / When 表示する / Then その旨が案内される

### US-021: レコード一覧をページングで見る
**ユウキ** として、テーブルを選んでレコード一覧をページング付きで見たい。

- Given READ 以上のテーブル / When 選択する / Then READ 以上のカラムのみを列とするレコード一覧がページング付きで表示される
- Given 実効権限が NONE のカラム / When 一覧を表示する / Then そのカラムは完全に非表示(列自体が存在しない。マスク表示ではない)— ユーザ確認済み(2026-07-18)

### US-022: レコードを絞り込む・並べ替える
**ユウキ** として、フィルタ条件とソートを UI で指定したい。SQL を書かずに目的のレコードへたどり着くためだ。

- Given レコード一覧 / When READ 以上のカラムに対しフィルタ・ソートを指定する / Then 条件が適用された一覧が表示される
- Given フィルタ指定 UI / When カラム候補を表示する / Then READ 以上のカラムのみが選択肢に出る

### US-023: WHERE / ORDER BY を手入力する
**ユウキ** として、WHERE 句・ORDER BY 句を直接書いて絞り込みたい。UI では表現しにくい条件を使うためだ。

- Given レコード一覧 / When WHERE / ORDER BY を手入力して適用する / Then 条件が適用される(この手入力はカラム権限に関係なく利用可能 — project-overview.md §5.4)
- Given 実効権限が NONE のカラム / When WHERE / ORDER BY 条件に指定する / Then 条件として使用できる(値の推測が可能になるサイドチャネルは**文書化されたリスク受容** — D-19。結果に NONE カラムの値そのものは表示されない)
- Given 構文エラーのある入力 / When 適用する / Then エラーが通知され、内部詳細(スタックトレース等)は表示されない

### US-024: レコードを編集する
**ユウキ** として、一覧上でレコードの値を編集したい。

- Given UPDATE 権限のあるカラム / When 一覧上で値を編集する / Then 変更が画面上に保留され、未反映であることが視覚的にわかる
- Given READ のみのカラム / When 編集を試みる / Then 編集できない
- Given 主キーを持たないテーブル / When 編集を試みる / Then 編集不可(D-15: 読み取り + 作成のみ)

### US-025: レコードを作成する
**ユウキ** として、新しいレコードを作成したい。

- Given US-014 の作成可否判定を満たすテーブル / When 新規行を追加し値を入力する / Then 作成が画面上に保留される
- Given 判定を満たさないテーブル / When 作成を試みる / Then 作成操作が提供されない

### US-026: レコードを削除する
**ユウキ** として、不要なレコードを削除したい。

- Given US-014 の削除可否判定を満たすテーブル / When 行を削除指定する / Then 削除が画面上に保留される
- Given 主キーを持たないテーブル / When 削除を試みる / Then 削除操作が提供されない

### US-027: 変更を一括反映する
**ユウキ** として、保留中の作成・更新・削除を「反映」ボタンで一括確定したい。中途半端な状態を作らないためだ。

- Given 保留中の変更(作成・更新・削除の混在可)/ When 反映する / Then 全変更が統一 API に単一トランザクションとして送信され、成功時は全件が確定する
- Given バッチ中の 1 件でも権限チェックまたは制約チェックに失敗 / When 反映する / Then DB への変更はゼロ件(オールオアナッシング)で、失敗理由が通知される
- Given 反映の実行 / When 完了する / Then データ更新操作として監査ログに記録される

## Epic 9: クエリビルダー(FR-09)— Phase 4

### US-028: タブ構成で SQL を組み立てる
**ユウキ** として、SELECT / FROM / JOIN / WHERE / GROUP BY / HAVING / ORDER BY / LIMIT OFFSET のタブで SQL を組み立てたい。SQL に不慣れでも参照クエリを作れるためだ。

- Given クエリビルダー画面 / When FROM / JOIN タブでテーブルとエイリアス(任意)を指定する / Then 選択済みテーブルが他タブから参照可能になる
- Given 選択済みテーブル / When 他タブでカラムを選ぶ / Then アクセス可能なカラムのみが候補に表示される
- Given SELECT / HAVING / ORDER BY タブ / When 集計関数を指定する / Then 集計関数を含む式を構成できる

### US-029: スキーマ非修飾の SQL を生成する
**ユウキ** として、組み立てた内容からスキーマ名を含まない SQL を生成したい。異なるスキーマで同じ SQL を使い回すためだ。

- Given タブでの指定内容 / When SQL を生成する / Then スキーマ非修飾の SQL(例: `SELECT * FROM customers`)が生成される
- Given 生成された SQL / When 保存または実行を選ぶ / Then クエリ保存(FR-10)・クエリ実行(FR-11)に引き継がれる

### US-030: SQL をクエリビルダーに読み込む(リバースエンジニアリング)
**ユウキ** として、既存の SQL をクエリビルダーの各タブに展開したい。過去のクエリを土台に編集するためだ。

- Given 他機能(履歴・保存クエリ等)からの遷移 / When SQL を渡す / Then SQL が解析され各タブ(SELECT / FROM / JOIN / WHERE / GROUP BY / HAVING / ORDER BY / LIMIT OFFSET)に反映される
- Given 解析できない SQL / When 渡す / Then その旨が通知され、元の SQL は失われない

## Epic 10: クエリ保存(FR-10)— Phase 4

### US-031: クエリに名前を付けて保存する
**ユウキ** として、SQL に名前を付けて保存したい。定型参照を再利用するためだ。

- Given 手入力またはクエリビルダー由来の SQL / When 名前と公開範囲(Public / Private)を指定して保存する / Then スキーマ情報を持たない形で保存される
- Given Public の保存クエリ / When 他ユーザが一覧を見る / Then 表示される。Private は作成者のみに表示される

### US-032: 保存クエリを編集する
**ユウキ** として、自分が作成した保存クエリを編集したい。

- Given 自分が作成した保存クエリ / When 編集する / Then 変更が保存される
- Given 他ユーザが作成した保存クエリ / When 編集を試みる / Then 編集できない(作成者のみ編集可能)

### US-033: 保存クエリを非表示にする(retired)
**ユウキ** として、使わなくなった保存クエリを論理的に非表示にしたい。一覧を汚さないためだ。

- Given 自分の保存クエリ / When retired にする / Then 通常の一覧に表示されなくなる(物理削除はされない)

## Epic 11: クエリ実行(FR-11)— Phase 4

### US-034: SQL を入力して実行する
**ユウキ** として、実行画面で SQL を入力・編集して実行したい。

- Given 実行画面 / When 読み取り専用 SQL を実行する / Then 結果が表形式で表示される
- Given 更新系 SQL(INSERT / UPDATE / DELETE / DDL 等)/ When 実行する / Then 実行が拒否される
- Given 実行 / When 完了する / Then SQL・パラメータ・対象スキーマ・結果件数・実行時間・実行日時・実行者が履歴に記録される

### US-035: パラメータ付きクエリを実行する
**ユウキ** として、`:param` 形式のパラメータを使ったクエリを実行したい。値だけ変えて安全に再実行するためだ。

- Given `:param` を含む SQL / When 実行画面に入力する / Then パラメータが自動検出され、値の入力 UI が表示される
- Given パラメータ値の入力 / When 実行する / Then NamedParameterJdbcTemplate によりバインドされて実行される(文字列連結は行われない)

### US-036: 実行時にスキーマを指定する
**ユウキ** として、SQL 実行時に対象スキーマを選択したい。同じ SQL を複数スキーマで使い回すためだ。

- Given 実行画面 / When スキーマ候補を表示する / Then 対象接続内で自分がアクセス権限を持つスキーマのみが選択肢に出る(許可リスト方式。D-16: 権限検証はスキーマレベルのみ)
- Given PostgreSQL / H2 / When 実行する / Then 実行直前に方言に応じたスキーマ切替(`SET search_path` 等)が行われる
- Given MySQL / MariaDB / When 実行する / Then 接続時点のカタログが使われ、追加のスキーマ切替は行われない

### US-037: 結果をページングで見る
**ユウキ** として、実行結果を必要に応じてページングして見たい。

- Given 実行設定 / When ページングの有無を指定して実行する / Then 指定に応じて結果がページング表示される
- Given 大量データ取得(閾値: 設定可能、デフォルト 100 件以上)/ When 実行する / Then 監査ログに記録される

### US-038: 保存クエリを実行する
**ユウキ** として、保存済みクエリを選んで実行したい。

- Given 保存クエリ / When 実行する / Then SQL は編集不可の状態で実行され、実行回数が履歴に記録される

## Epic 12: クエリ履歴(FR-12)— Phase 4

### US-039: 実行履歴を一覧で見る
**ユウキ** として、クエリ実行履歴をページング付きで見たい。過去の作業を振り返るためだ。

- Given 履歴画面 / When 一覧を表示する / Then 保存クエリ実行と直接入力実行が区別されて表示される

### US-040: 履歴を絞り込む
**ユウキ** として、履歴を条件で絞り込みたい。

- Given 履歴一覧 / When 実行日時(範囲)・実行者(全ユーザ/自分のみ)・対象スキーマ・SQL テキストで絞り込む / Then 条件に合致する履歴のみ表示される

### US-041: 履歴から他機能へ遷移する
**ユウキ** として、履歴のクエリをクエリ実行・クエリ保存・クエリビルダーに引き継ぎたい。

- Given 履歴エントリ / When 遷移先を選ぶ / Then SQL(とパラメータ)が引き継がれた状態で遷移する(クエリビルダーへは US-030 の解析を経由)

## Epic 13: 監査ログ(FR-13)— Phase 1〜4(閲覧は Phase 4)

> **記録基盤の位置づけ(D-20)**: 監査ログの記録機能(記録 API・基盤)は全機能から呼ばれるため **Phase 1 で整備**する。記録は主処理とは**別トランザクション**で行い、主処理が失敗(例外発生)した場合もその事実を含めて記録が残る。閲覧画面(US-046)のみ Phase 4。

### US-042: 認証イベントが記録される(Phase 1)
**アキラ** として、ログイン・ログアウト・ログイン失敗が記録されてほしい。

- Given 認証イベント / When 発生する / Then 日時・ユーザ名(失敗時は試行ユーザ名)が内部 DB に構造化形式で記録される(IP アドレス・セッション継続時間はスコープ外)
- Given 主処理が失敗(例外発生)した操作 / When 監査ログを記録する / Then 記録は主処理と別トランザクションのため、主処理のロールバックに巻き込まれず失敗の事実が記録される(D-20)

### US-043: 管理操作が記録される(Phase 2)
**アキラ** として、管理操作の履歴が残ってほしい。

- Given アカウント登録申請・承認/却下・接続設定変更・スキーマ取込・権限設定変更・グループ変更・YAML エクスポート/インポート / When 実行される / Then それぞれ独立したイベント種別として記録される
- 記録項目: 日時(ISO 8601)・ユーザ ID・対象接続 ID・操作種別・対象リソース・結果ステータス(イベント種別コードで記録し、表示時に UI 言語で解決 — H-07)

### US-044: データアクセスイベントが記録される(Phase 3〜4)
**アキラ** として、データ操作の証跡が残ってほしい。

- Given 大量データ取得(デフォルト 100 件以上、設定可能)・クエリ実行(結果件数・実行時間含む)・データ更新操作(挿入/更新/削除)/ When 発生する / Then 監査ログに記録される

### US-045: セキュリティイベントのメール通知を受け取る(Phase 1、D-13)
**アキラ** として、連続ログイン失敗などのセキュリティイベントが閾値を超えたらメールで知りたい。異変に早く気づくためだ。

- Given セキュリティイベント(連続ログイン失敗・認可違反等)が閾値(設定可能)を超過 / When 検知される / Then 管理者へメールが送信される(SECURITY-14 対応)

### US-046: 監査ログを閲覧する(Phase 4)
**アキラ** として、監査ログを画面で閲覧したい。

- Given 管理者 / When 監査ログ画面を開く / Then ログをページング付きで閲覧できる
- Given 一般ユーザ / When 監査ログ画面へアクセスを試みる / Then 拒否される(管理者のみ閲覧可能)

## Epic 14: UI 基盤(NFR-06 / NFR-08)— Phase 1

### US-047: UI 言語を切り替える(D-09)
**ユウキ** として、UI の言語を日本語と英語で切り替えたい。

- Given 任意の画面 / When 言語を切り替える / Then UI 文言が選択言語で表示され、設定が保持される
- Given 登録申請(US-001)/ When 登録確認メールを送信する / Then 申請時の UI 言語を引き継いだ言語で送信される(ユーザ確認済み 2026-07-18)
- Given 登録済みユーザへのメール送信(承認結果等)/ When 送信される / Then 受信者の言語設定に応じた言語で送信される

- Given 初回アクセス / When 表示される / Then OS のテーマ設定に連動したテーマで表示される
- Given 任意の画面 / When 手動でテーマを切り替える / Then 選択が OS 設定より優先され、保持される
